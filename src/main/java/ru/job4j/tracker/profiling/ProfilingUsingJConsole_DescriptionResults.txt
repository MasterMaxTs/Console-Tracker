1. Как видно из результатов профилирования приложения memTracker, создание до 1_000_000 items (опционально с последующим
удалением) не является слишком затраным ресурсом относительно использования heap memory (среднее использование: 190Mb).
2. При создании 10_000_000 items (опционально с последующим удалением) использование heap memory возрастает вплоть
до 1.63Gb! При этом, количество чисток области Young Generation сборщиком G1 составило 19 с сумарным временем простоя
приложения 1,026sec.
Вывод на консоль, как видно из результатов, тоже использует дополнительную память:
- создание 10_000_000 items с их последующим консольным выводом Heap memory usage: 2.85Gb
(GC time: 1,158 seconds on G1 Young Generation (33 collections), 0,000 seconds on G1 Old Generation (0 collections));
- создание 20_000_000 items с их последующим консольным выводом Heap memory usage: 3.67Gb
(GC time: 2,200 seconds on G1 Young Generation (107 collections), 0,000 seconds on G1 Old Generation (0 collections));

Таким образом, если планируется использование приложения memTracker для работы с относительно не большим количеством
заявок (до 1_000_000 шт.), то можно ограничиться максимальным размером кучи 200Mb (-Xmx200M).
При создании большого количества заявок (10_000_000 - 20_000_000) размер кучи не может быть меньше 2.85Gb. В этом случае
можно ограничиться default максимальным размером кучи 4Gb.
